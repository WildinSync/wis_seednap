# teleo_rhone -- example outputs

A side-by-side look at what SeeDNAP produces for one dataset (`teleo_rhone`,
the Rhone 2025 teleo run), plus the FAIRe sample manifest. The first three
files are the pipeline's existing outputs; the fourth is generated by the
`seednap manifest` command.

> [!NOTE]
> These are **trimmed** copies committed so a fresh clone can see the shape of a
> finished run without shipping large files. Each CSV here is the real header
> plus the first few representative rows; a full run produces thousands of OTU
> rows and one row per sample. Row counts quoted below are from the full run,
> not from these excerpts.

| File | Where it comes from | One row per | What it tells you |
|---|---|---|---|
| `1_read_tracking.csv` | report step (`04_report/`) | **sample** (20) | reads surviving each stage: `raw -> trimmed -> clustered`, plus `pct_retained`. The data-loss audit. |
| `2_otu_table.csv` | SWARM step (`02_swarm/`) | **OTU** (2,645) | the cleaned abundance matrix: `sequence` + one column per sample, counts in cells. |
| `3_taxonomy_blast.csv` | taxonomy step | **OTU** (2,645) | the same matrix with taxonomy joined on the left: `ASV_ID, pident, kingdom...species, is_contaminant_candidate`, then the per-sample counts. The headline scientific table. |
| `4_sample_manifest_FAIRe.csv` | `seednap manifest` | **sample-library** (20) | the canonical, validated per-sample record: identity, geography, dates, the control type, the extraction batch, and the sequencing-run grouping -- see below. |

## What the FAIRe manifest adds

The first three files describe **OTUs and reads**. None of them carries, in a
standardized machine-readable form, *what each sample is*. That lived scattered
across the lab's `metadata_field_*.csv` / `metadata_proj_*.csv` (and, for older
runs, a demux `metadata_lab_*.csv`). The manifest consolidates that into one
canonical table -- one row per sample-library -- anchored on the **FAIRe v1.0.2**
eDNA metadata standard, keeping DwC `eventID` as the key. Columns include:

- **identity / provenance** -- `eventID`, `eventDate` (normalized to ISO-8601:
  the lab's `2025.08.19` -> `2025-08-19`), `decimalLatitude/Longitude`,
  `env_medium`, `samp_size` (from the lab's `volume`), `target_gene`.
- **controls (the two-field FAIRe model)** -- `samp_category`
  (`sample` / `negative control` / ...) + `neg_cont_type`. Here `Blank-ext-1` ->
  *extraction negative* carrying its `extraction_ID` (EXP524); `Blank-PCR-1` ->
  *PCR negative* with no `extraction_ID` (it cleans the whole dataset). This
  replaces the fragile name-substring guess used at GBIF-export time.
- **grouping** -- `seq_run_id` (the DADA2 error-model batch key) and
  `extraction_ID` (the control->sample association key).
- **per-stage QC counts** -- `reads_raw ... reads_nonchimeric` columns exist in the
  schema (pipeline-written; blank here because this manifest was migrated from
  metadata, not produced by a run).

## A real discrepancy this surfaces

Cross-checking the manifest against `2_otu_table.csv` flags one mismatch:
the OTU table has a sample column **`Blank-PCR-3`** with no matching metadata
row, while the metadata lists **`Blank-PCR-1`**. That is a genuine lab
bookkeeping inconsistency the validator catches up front (it would
otherwise be a silently-unlabelled control in the dataset).
